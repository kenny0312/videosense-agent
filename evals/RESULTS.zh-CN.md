# τ²-video 评测结果 — VideoSense

*[English version / 英文版](RESULTS.md)*

对 VideoSense（VS，大脑是 Gemini 的多轮多模态视频理解 agent）做的 τ²-bench 式评测。
被测的是**真 agent**（真 Gemini 进循环），所有后端都换成密封的假世界——不碰生产数据、结果可复现。

## 数据集

**145 道题**（单轮 + 多轮，47 道必过），金标严格对齐假片库（16 个视频、带阶段标注的跳伞片段、
约 50 条核实过的事实含负事实）。`python -m evals check` 零问题，并且会检查每个计分项都有对应
配置——防止一道配错的题静默拿 0 分。

| 方面 | 考什么 |
|---|---|
| 找视频 / 诚实 | 找对视频；真没有才说没有；宽类有的绝不能瞎说没有；负事实（比如确实**没**戴头盔） |
| 数量 / 时间点 | 计数、去重（DISTINCT 陷阱）、聚合；时序定位按重合度判 |
| 用对工具 / 展示 | 该播/该列表/该查库/该联网/该记偏好各就各位；无关请求婉拒；用 show_* 交付、id 不进文本 |
| 多轮 | 指代、省略、纠正、约束累积 |
| 双向控制 | 用户中途改共享状态（上传/入库/贴图），假世界真的跟着变 |
| 看画面 | 只能"看"才知道的问题（颜色、人数）；假片库没有真视频文件，analyze_video 按事实清单回放 |
| 安全 / 身份 / 自知 | 拒绝提示注入、不漏底层供应商、对没建的功能诚实说"拿不到" |

## 怎么判分

门禁只用**确定性判分器**（不用 AI 当裁判决定过没过）：工具审计、交付面查全+查准（答案 **加**
show_* 侧信道——对齐"原始 id 不进文本"的产品规则）、时间 IoU、精确计数、诚实/拒答检查、
id/供应商泄漏检测、多轮槽位跟踪、世界状态断言。每道题声明自己用哪几把尺子。
另有一个跨家族 Claude 裁判可以给开放式判据打**参考分**——但绝不进门禁，除非先和人工标注对过表。

## 真跑基线（真 Gemini，143 道计分题，每题 1 次）

**131 / 143 通过（92%）**，95% 波动区间约 86~95%（批⑤尺子校准后，2026-07-08）。
2 道环境故障已剔除，不算模型的锅。

| 方面（每把尺子在多少道题上得满分） | 通过 | 通过率 |
|---|---|---|
| 安全拒答 · 身份不漏底 · 时间点准 | 9/9 · 5/5 · 22/22 | 100% |
| 多轮不忘事 · 世界状态对了 | 29/29 · 9/9 | 100% |
| 工具用得对 | 116/121 | 96% |
| 该问就问（不越界代答） | 18/19 | 95% |
| 诚实不瞎编 | 30/32 | 94% |
| 数量对 | 33/36 | 92% |
| 不泄漏原始 id | 8/9 | 89% |
| 找对视频 | 29/34 | 85% |
| 实体对得上 | 4/5 | 80% |

必过题：46/47（失守的 1 道 toolcall-table-all-videos-23 疑似金标设计问题——"列表格"题用
"正文里说没说 16"当完整性代理，待下轮复核）。找视频仍是最真实的短板（见下）。

### 这个数字是怎么变可信的

两轮真跑，都是每题 1 次。第 1 轮 82%，第 2 轮修完**判分器**（不是改 agent）后升到 88%。
配对比较（只看翻转的题）算出这次提升 p≈0.04——小到可以说是真变化，不是运气。
就是这个循环——跑 → 把每个失败归类成 *真缺陷 / 金标错 / 判分器错 / 环境噪声* → 修工具 → 再跑——
把一个粗原型变成了一把可信的尺子。修好的判分器都冻进了"金答案回放"单测，防止再退回去。

### 批⑤冤案清理（2026-07-08）：88% → 92%（尺子校准，不是 agent 变好）

对 88% 基线的 17 道失败做了一次多智能体逐案复核（44 个复核员/反方律师/审计员，
每个"冤案"指控要过两名反方对抗验证）：**7 道是冤案**（判分器 bug 或金标错——包括一道
判分器不认 `a|b` 写法导致怎么答都挂的必过题），**10 道维持真缺陷**。修复全部落在评测侧
（尺子 6 处 + 金标 9 题 + 出题 lint 2 条 + 失败集中度报警），修正后重跑：**131/143（92%）**，
7 道平反题全部转绿，真缺陷保持红。这 4 个百分点是"尺子量准了"，不是 agent 变好了——
新旧分数的尺子指纹不同，不做跨比。复盘的制度教训：**改判分原则必须扫全套件**（原则只修
触发它的那道题，孪生题就成了下一轮的冤案）；**失败大量堆在同一把尺子上，先查评测侧**。

## 评测抓到的真缺陷（登记在案，按会话分工暂不顺手修）

1. 🔴 **空白答案**（新发现，产品级 bug）：Gemini API 安全拦截时 `loop_driver` 把空响应原样透传，
   用户拿到一片空白而不是拒答——本轮随机砸中 3 道题（count-makeup / vision-oven / upload-honesty），
   上轮砸中 1 道，是当前最大的随机失败源。修法已定位：`GenAIConversation.send` 读 finish_reason +
   `orchestrator.py` 对空答案兜底。
2. 🟠 **检索召回靠印象、不兜底**：场景/品类类问题只按猜的关键词滤谓词，不查标题、不扫全库（才 16 条），
   漏掉够得着的正确答案——park / outdoor-scenery / tutorial / fastpaced 几道题跑一次闪一次。
3. 🟠 **提示注入 & 身份漏底**：已改提示词修复并真跑验证转绿；必过题当回归守卫。
4. 🟡 **偶发 id 进答案文本**：深度对比题把 `sky01` 写进正文（产品规则要求 id 走侧信道）。
5. 🟡 **越界代答**：问天气真去 web_search 了，违反自己的工具边界说明。
6. 🟡 **花费自知**：一个还没建的功能；agent 诚实说拿不到这数（评测现在把这算作正确，等功能上线再改口径）。

## 复现

```bash
python -m evals check      # 校验金标 + 配置
python -m evals            # 免费脚本车道 + 本地仪表盘
python -m evals live       # 真 Gemini（要 GCP 凭证，花 token）
python -m evals serve      # 本地控制台：点按钮跑、自动刷新
```

仪表盘（双语，`dashboard.html` / `dashboard.en.html`）能看趋势、各方面对比、和上次比新过/新挂的题，
以及每道失败题点开即下钻的卡片（题目 / 答案 / 期望 / 工具链 / 金标依据）。看结果完全不用 push GitHub。
