# 设计:视频档案 v2(入库信息的目标形态)

> 状态:Proposal(待审)· 起因:2026-07 三场审计的合订本 —— facts 真值抽检
> (谓词精度 ~85%、时间戳错位 ~18%、confidence 与真伪不相关)、D3 检索重标定
> (transcript 裸段是枢纽噪声主产地)、D2 验收(383 个视频标题=id 空壳)。

## 0. 设计纪律(先于字段)

**每个字段默认是负债,直到通过"三问"证明是资产:**

1. **消费者**:哪个查询/哪道考题/哪块 UI 用它?(点名,不许"以后可能有用")
2. **审计口径**:它错了,靠什么发现?(没有审计手段的字段 = 埋给下游的地雷,confidence 之鉴)
3. **过期归属**:谁负责刷新?(没有 owner 的数据必然漂移,skydive 竖井之鉴)

本设计**只加 3 个字段、升级 2 处、降级 1 个、明确 1 层为衍生品**;
物体清单/人数/情绪基调等诱人 feature 一律不进 —— 等真实查询日志投票(§6)。

## 1. 身份层(video_metadata,一行/视频)

| 字段 | 状态 | 三问摘要 |
|---|---|---|
| `video_id` | 存 | 主键 |
| `title` | **升级** | 383 个空壳(=id)由 caption 生成一句人类标题回填;新入库由 enrichment 顺产。消费者=检索别名/前端/报告;审计=抽检可读性;owner=enrichment |
| `gcs_uri` `duration_sec` | 存 | — |
| `source` | **新** | 枚举 activitynet/pexels/user_upload。消费者=配额与合规策略、语料统计;审计=与 id 前缀一致性校验;owner=入库脚本 |
| `ingested_at` | **新** | 消费者="最近添加"类查询、增量重建;审计=非空约束;owner=入库脚本 |

## 2. 理解层(video_facts,~4-8 行/视频)

| 字段 | 状态 | 说明 |
|---|---|---|
| `predicate` `matched` `rationale` | 存 | 含 matched=0 的"已核对无"负例(诚实回答的依据) |
| `start_ts` `end_ts` | **升级** | 重抽取协议改为**证据先行**:提示词强制"先描述你在第 X 秒看到的画面,再据此给区间"——时间段是证据的推论,不许裸报(治 18% 锚在片头的错位)。审计=真值抽检脚本(已建)按"该时刻真有该内容"口径验收,错位率 <5% 批次才算完成;消费者=跳转到证据时刻(产品差异化点)+ timestamp 评测维度 |
| `verified` | **新** | 三态:unverified / machine(真值审计通过)/ human。消费者=检索加权与审计报表;审计=就是审计本身;owner=审计批次回写 |
| `confidence` | **降级** | 字段保留(免迁移),**信号弃用** —— 实测(2026-07-13,42 样本+复审)与真伪不相关(低档 71% 真 vs 高档 69%)。下游禁止当权重;由 `verified` 接任质量信号 |

## 3. 检索层(content_embeddings,~8-15 行/视频)

| 行类型 | 状态 | 内容形态 |
|---|---|---|
| `video` 粗向量 | **新** | 每视频 1 条:「标题 + caption + 活动词」合成嵌入。消费者=两级检索第一级(索引 >10k 行时的必需品,现在是提前量)+ "这个视频是什么"的规范摘要;审计=探针 hit 指标;owner=enrichment |
| `caption` | 存 | 整片视觉描述(眼睛通道:场景/动作/物体) |
| `transcript` | **升级** | Contextual 化:每段前置出身说明再嵌 ——「视频《清晨湖钓》(钓鱼) 第2段: 好,看我手腕动作」。依据:裸口语段(2081 行,索引最大来源)是枢纽噪声主产地(D3 探针实录);Anthropic Contextual Retrieval 同型技术官方实测检索失败率 -35~49% |
| `fact` | 存 | "谓词: 理由"格式天然自带上下文,不动 |
| `analyze` | 存 | 付费观看问答缓存(用到哪长到哪;耳朵+眼睛的付费精读) |

**caption 与 transcript 不冗余**:眼睛与耳朵两个通道(无声风景片只有 caption 能描述,
教学干货全在 transcript 嘴上),粒度也不同(整片 vs 逐段带时间戳)。
真正的冗余在查询端 —— 同一视频的多种行在 top-k 互相抢名额,修法是
**查询端按视频聚合取最高分**(检索层任务,不是删数据)。

## 4. 衍生层(可随时幂等重建,非事实源)

- `all_activities` / `category`:从 matched facts 过受控词表生成,专供 SQL 计数/筛选车道。
  **约定写死:与 facts 冲突时以 facts 为准**;重建脚本一键刷新(已具备,本设计只是把约定成文)。
- `skydive_segments` 竖井:历史遗留的结构性重合,15 视频量级 —— 登记不动;
  唯一纪律:**不再新建第二个专项竖井**(要专项信息就进 facts/检索层)。

## 5. 一个视频的 v2 画像(示例)

```
v_abc123 |「清晨湖边钓鱼实拍」| 92s | activitynet | 2026-07-14
  facts: 钓鱼✓[5-88s,machine] · 抛竿✓[12-18s,machine] · 划船✗(已核对无)
  检索行: 1 video 粗向量 + 1 caption + 6 contextual transcript 段 + 3 fact + 0 analyze
  衍生: activities=[fishing, casting line] · category=outdoor
```

## 6. 明确不做(等证据)

物体清单、人数、场景结构化、情绪/节奏标签 —— 每一个都要先过三问的第一问:
**挖真实查询日志,看用户真问什么再喂什么**。每加一类 feature 就多一处要审计的质量面。

## 7. 迁移(全部幂等、可分批、不改线上行为)

| 批 | 内容 | 成本 | 验收 |
|---|---|---|---|
| M1 入库质量小批 | title 回填 + source/ingested_at 加列回填 + transcript contextual 重嵌 + video 粗向量 | ~$1.5,半天 | D3 探针套件复跑:中文 hit@3 不降、缺席拦截不降;标题抽检可读 |
| M2 时间戳批 | facts 证据先行重抽 + verified 回写 | ~$3-5 | 真值审计错位率 <5% |
| M3 检索端 | 查询按视频聚合 + (索引>10k 时)两级检索启用 | 代码,无数据成本 | 评测检索维度不降 |
