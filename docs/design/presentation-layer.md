# 设计:呈现层 —— 让输出更"高级"(不违背"别多给")

> 状态:Design(评审后动手) · 范围:`pipeline/loop_driver.py`(收口)、`web/index.html`(渲染)、可选 `pipeline/present.py`(新) · 关联:Pandora 对照测(用户反馈"Pandora 输出更高级")、[program-memory-and-lessons](../../.claude/...)、prompt-keep-adaptive

## 1. 问题定性(诚实版)

用户反馈:Pandora 输出显得更"高级"。测试证明:**不是 Pandora 工具更多/能力更强**(它核心搜索是坏的,VS 论 substance 全面更好);而是 **VS 的"产出"比 Pandora 少**:

1. **VS 最终答案 = 大脑一次性写的文本,没有专门的"呈现/组织"环节** —— 直、朴素、利落(这是旧原则「别多给、问什么答什么」用力过猛的另一面)。
2. **缺一致的结构** —— 有时列点、有时一段话,视模型心情。
3. 富数据(视频卡/表格/marker)在侧信道有,但**包裹它的文字是白开水**。

"高级感" = 呈现结构 + 轻综合 + 一致性,**不是答得更多**。核心张力:**既要更精致,又不能违背「别多给」**。解法不是"多答",是"同样的内容,组织得更好"。

## 2. 目标 / 非目标

**目标**:同一份答案内容,收口时【结构化 + 一致】呈现 —— 让它读起来是"打磨过的",而非模型随手写的。
**非目标**:
- 不多答用户没问的(「别多给」红线不破);
- 不加一次额外 LLM 调用做"润色"(贵、慢、且是又一层);
- 不搞花哨动效/emoji 堆砌(前端刚 refresh 过,克制)。

## 3. 三个候选方案(评审选一/组合)

### 方案 A:收口格式契约(prompt 层,最轻,零成本)⚡
在宪法/教训里加一条**收口呈现规则**(不是多答,是"答得有结构"):
- 多条结果 → 用带**内容标签**的编号列表(「第 1 个,橙色跳伞服出舱那个 · 1:46」),不是一坨;
- 有结论 → **一句话结论先行**,再列依据(倒金字塔);
- 数字 → 关键数字**加粗**、对账句式;
- 语言、粗细一致(承接刚修的 drift guard)。
- 成本:0(纯 prompt)。风险:模型不总遵守 → 需探针验一致性。

### 方案 B:结构化收口信封(代码层,中等)
让大脑收口时【也】吐一个**轻结构**(不是纯文本):`{headline, points[], footnote}`,前端按固定模板渲染(标题行 + 要点 + 脚注)。
- 好处:呈现**由前端模板保证一致**,不靠模型自觉;富数据(视频/表格)天然挂在结构上。
- 代价:改收口协议 + 前端渲染 + 大脑要学会吐这个结构(function-calling 或 JSON 收口)。是"又一层",但很薄。

### 方案 C:前端 markdown 增强(纯前端,最省心)
不改后端,只把前端的答案渲染从"纯文本"升级为**完整 markdown**(标题/列表/加粗/表格/引用块)+ 排版打磨(行距、层级、代码块)。模型本来就常写 markdown,现在真正渲染出来。
- 好处:零后端风险、纯观感提升、立刻见效;
- 局限:治标(结构还是靠模型写),但配合方案 A 的收口规则就够。

## 4. 建议(供拍板)

**A + C 组合,先不做 B**:
- **A(收口格式契约)**:一条收口规则统一结构(结论先行、带标签编号、关键数字加粗、语言一致)——「组织得好」不等于「答得多」,不破红线;
- **C(前端 markdown 增强)**:把这些结构真正渲染出来(现在前端对 markdown 支持有限);
- **B 暂缓**:结构化信封是"又一层",等 A+C 实测不够精致了再上 —— 符合「别加层」。

预期效果:同样的"有,14 个跳伞视频",从一句白话 → 变成「**14 个**跳伞视频 · 结论行 + 编号带内容标签 + 时间码可点」的打磨版,但**一个字没多答**。

## 5. 验收

- 探针:重跑 Pandora 对照测的 E 维(多轮展示)+ 随机 5 问,人工比对"呈现精致度"(结构一致性、结论先行率、无语言漂移);
- 不回退:内容正确性、侧信道视频/表格、成本不变(A 零成本、C 纯前端);
- A/B 观感:新旧两版截图对比给 owner。

## 6. 开放问题(评审定夺)

1. **A vs A+C vs A+B+C**?(倾向 A+C —— 最大观感/最小风险;B 是"又一层",留触发式)
2. **结论先行**是否对【所有】回答强制,还是只对多结果/分析类?(倾向:只对"有多条结果/有判断"的,单值问答如「有几个」保持一句话,别为形式加长)
3. **markdown 渲染**用现成轻库(marked.js,~10KB)还是手写最小解析?(倾向:CDN marked.js,别造轮子;CSP 允许 cdnjs)
