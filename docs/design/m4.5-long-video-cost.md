# M4.5:长视频提速 + 成本收尾(time-range 硬裁剪 + flash 两段式粗筛)

> 状态:Design(评审后即动手) · 范围:`perception/analyze_video_contextual.py`、`pipeline/node_executor.py`、`pipeline/node_specs.py`、`pipeline/loop_driver.py`(prompt) · 关联:M4(并行+缓存已上线)、`realtime-video-understanding.md` §9 #3/#4(本文补完这两项)

## 1. 背景 / 问题
M4 把成本清单 §9 的 #1(top-K 引导)/#2(并行)/#5(缓存)做了。**剩 #3(flash 两段式)、#4(长视频 time-range 采样)没做** —— 而 M4.3 实测暴露它们正是当前最痛的:
- **长视频慢且贵**:同一视频实测单次 **64–182s / ~60k token**。现在 `analyze_video` 把【整段】视频喂给 Gemini,哪怕只关心其中 30s。
- **多候选浪费**:N 个候选全量细看 = N 次昂贵多模态;缺一个【便宜的先筛一遍】。

## 2. 目标 / 非目标
**目标**
1. **time-range 硬裁剪**:`analyze_video` 能只看视频的【某个时间段】—— 真的让 Gemini 只处理该段(省 token + 提速),不是现在的"prompt 软提示"。
2. **flash 两段式粗筛**:多候选时,先用便宜的【短片段/粗看】筛出 top-K,再细看 —— 复用 #1 的能力。

**非目标**
- 不上 ffmpeg 预裁(spike 证明 Gemini 原生支持 video 偏移,见 §3)。
- 不迁移到 `google-genai` SDK(vertexai 1.149 的 `_raw_part` 已够,见 §3)。
- 不做自动"精彩段定位"模型(Phase 2 仅靠 brain + 短片段 triage;真·自动定位留后续)。

## 3. 现状 / spike(已核实)
- `AnalyzeRequest.time_range` 字段【存在】,但:① `_run_analyze_video` 现在**根本没把它从 node.inputs 透传**进去;② 即便传了也只是 `build_prompt` 的**软提示**(整段视频仍被 ingest)→ 不省钱不提速。
- **SDK 能力(spike 结论)**:
  - `vertexai` 1.149 的 `Part.from_uri(uri, mime_type)` **无**高层裁剪参数;
  - 但 `Part._raw_part.video_metadata` 这个底层 proto **有 `start_offset` / `end_offset`**,实测可设、`to_dict()` 带得出去 → **Gemini 只处理该时间段**。无需 ffmpeg、无需换 SDK。
  - (`google-genai` 1.74 也已装、原生支持裁剪 + FPS,但用它要迁移 analyze 的整条 Gemini 调用 —— 本期不做,留作未来若要 FPS 采样再议。)

## 4. 设计

### 4.1 Phase 1:time-range 硬裁剪(基础,高 ROI)
**① 透传 + 硬裁剪**
- `node_specs` 的 `analyze_video` 参数加 `time_range`(`[start_sec, end_sec]`,可选)。
- `_run_analyze_video` 把 `node.inputs["time_range"]` 解析进 `AnalyzeRequest.time_range`。
- `_gemini_generate`:若 `time_range`,在 `Part` 上设 `_raw_part.video_metadata.start_offset/end_offset`(`duration_pb2.Duration(seconds=…)`)→ Gemini 只看该段。
- `build_prompt` 仍把 time_range 写进 prompt 文字(双保险:模型既被裁剪、也被告知"聚焦这段")。

**② 谁来定这个段**
- **brain 主动设**:用户/上下文暗示了段落("看结尾"/"第 2 分钟那段"/"跳伞开伞后")→ loop 大脑给 `time_range`。prompt 加一句引导。
- **skydive 已有锚点**:`skydive_segments` 表已有各阶段 ts → 查到后喂给 `analyze_video` 的 time_range(skydive 专栏直接受益)。
- **缓存键已含 time_range**(M4.1 已经把它放进 key)→ 不同段分别缓存,天然正确。

> 收益:只看 30s 而非 10min → token 和延迟按比例降;长视频不再是 182s 的拖累。

### 4.2 Phase 2:flash 两段式粗筛(orchestration,建立在 Phase 1 上)
多候选时分两段:
1. **粗筛(便宜)**:对每个候选发 `analyze_video`,问一个**terse 相关性问题**("这视频和 X 相关吗?0–1")+ 用 **time_range 只截开头几秒** → 又快又省 → 拿到相关性分。
2. **细看(贵)**:只对 top-K(2–3)发完整 `analyze_video`(必要时 pro 复核)。

落地方式(择一,评审定):
- **(a) 纯 prompt 引导**:`_LOOP_SYSTEM` 教大脑"候选多→先短片段 terse 粗筛→再细看 top-K"。最简、零新机制,靠 Phase 1 的 time_range 就能跑。**推荐先走 (a)。**
- **(b) 半结构化**:给 `analyze_video` 加个 `mode: triage|full`,triage 自动用短 clip + terse prompt + 强制 flash。更可控但加面。

## 5. 改动点 + 测试
| 文件 | 改动 |
|---|---|
| `perception/analyze_video_contextual.py` | `_gemini_generate` 接受/设置 video offset;`build_prompt` 已含 time_range 文字(确认) |
| `pipeline/node_executor.py`(`_run_analyze_video`) | 把 `inputs.time_range` 透传进 `AnalyzeRequest` |
| `pipeline/node_specs.py` | `analyze_video` 参数加 `time_range` |
| `pipeline/loop_driver.py`(`_LOOP_SYSTEM`) | 引导:有段落线索就给 time_range;多候选先短片段粗筛 |

**测试**
- 单测:`_gemini_generate` 在给 time_range 时,提交的 Part 带 `video_metadata.start/end_offset`(mock generate,断言 Part proto);不给则不带。
- 单测:`_run_analyze_video` 透传 time_range;缓存键随 time_range 变(M4.1 已覆盖一半,补一条)。
- 实测:同一长视频【全段 vs 截 30s】的 token / 墙钟对比(应显著下降);skydive 段定位→只看该段。

## 6. 里程碑(小步可回退)
- **M4.5a time-range 硬裁剪**(Phase 1):透传 + `_raw_part.video_metadata` + prompt 引导。**独立高价值、可单独上线**;不裁剪(不传 time_range)= 现状,零回退成本。
- **M4.5b flash 两段式**(Phase 2):先走 (a) 纯 prompt 引导,跑一阵看大脑会不会用;不够再上 (b) `mode` 参数。

## 7. 开放问题(评审定夺)
1. **段落由谁定**:Phase 1 先只支持"brain/skydive 显式给 time_range";要不要现在就做一个"先 flash 粗定位精彩段"的自动 locate(更强但更贵、更复杂)?倾向**先不做**。
2. **粗筛用 (a) prompt 还是 (b) mode 参数**:倾向先 (a),靠 Phase 1 + 引导,观察大脑是否自然两段式。
3. **`_raw_part` 是半私有属性**:vertexai 1.149 已 pin 且在被弃用(API 不会再大改),可接受;但要留注释 + 一条"升级 SDK 时复查"的提醒。是否值得借此机会迁 `google-genai`(更干净 + FPS 采样)?倾向**本期不迁**,单列未来项。
